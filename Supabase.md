# Supabase & Local Development Guide

This document explains how to integrate Supabase into your production environment and how to manage your local PostgreSQL development setup.

## 1. Production Setup: Connecting Supabase to Vercel

Since you are migrating from Neon to Supabase, you can clean up your Vercel environment variables and focus on a single connection string.

### Required Environment Variables on Vercel
To make Supabase work with your production code, you only need to set these in your Vercel dashboard:

| Variable | Value |
| :--- | :--- |
| `DATABASE_URL` | Your Supabase **URI** (Transaction mode recommended) |
| `DATABASE_SSL` | `true` |

### Steps to Integrate:
1.  **Get Connection String**: In Supabase Dashboard, go to **Settings > Database > Connection string > URI**.
2.  **Update Vercel**: Replace the existing `DATABASE_URL` in Vercel with the Supabase URI.
3.  **Initialize Schema**: If you haven't yet, run the DDL against your Supabase instance:
    ```bash
    psql "YOUR_SUPABASE_URI" -f backend/db/DDL.sql
    ```

## 2. Local Development Setup (pgAdmin)

Your development environment is set up to run on a local PostgreSQL instance, which you can manage via **pgAdmin**.

### Local Configuration (`.env.development`)
I have already configured your local setup to use:
-   **Database**: `lumina_dev`
-   **URL**: `postgresql://localhost:5432/lumina_dev`

### How to Access via pgAdmin:
1.  **Open pgAdmin 4**.
2.  **Add New Server**:
    -   **General Tab**: Name it "Lumina Local".
    -   **Connection Tab**:
        -   **Host**: `localhost`
        -   **Port**: `5432`
        -   **Maintenance database**: `postgres`
        -   **Username**: `nr` (or your system username)
        -   **Password**: (Leave blank or use your local Postgres password)
3.  **Browse Tables**: Once connected, navigate to `Databases > lumina_dev > Schemas > public > Tables`.

## 3. Environment Variable Decluttering (Vercel)

You can safely remove the following variables from your Vercel dashboard as they are either auto-generated by the old Neon integration or redundant.

### ❌ Remove (Neon Redundant)
- `POSTGRES_DATABASE`, `POSTGRES_HOST`, `POSTGRES_PASSWORD`, `POSTGRES_USER`, `POSTGRES_URL`, `POSTGRES_URL_NON_POOLING`, `POSTGRES_URL_NO_SSL`, `POSTGRES_PRISMA_URL`
- `PGDATABASE`, `PGHOST`, `PGPASSWORD`, `PGUSER`, `PGHOST_UNPOOLED`
- `DATABASE_URL_UNPOOLED`
- `NEON_AUTH_BASE_URL`, `NEON_PROJECT_ID`, `VITE_NEON_AUTH_URL`

### ❌ Remove (Unused in Code)
- `GOOGLE_CLIENT_SECRET` (Lumina uses frontend-only OAuth or backend ID validation which doesn't require the secret in the current implementation)
- `PROJECT_NAME`, `PROJECT_NUMBER`

### ✅ Keep (Required)
- `DATABASE_URL` (Update this to Supabase)
- `DATABASE_SSL`
- `JWT_SECRET`, `JWT_REFRESH_SECRET`, `JWT_ACCESS_EXPIRES_IN`
- `FRONTEND_URL`
- `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASSWORD`, `SMTP_FROM_EMAIL`
- `GOOGLE_CLIENT_ID`, `VITE_GOOGLE_CLIENT_ID`
- `API_KEY` or `GEMINI_API_KEY`
- `VITE_API_URL`, `VITE_API_PREFIX`
- `NODE_ENV`
